
# Project description

Real-Time Chat Application | Kafka, Spring Boot, WebSocket, React

Designed and implemented an event-driven chat system using Apache Kafka for asynchronous message processing and WebSocket (STOMP) for real-time communication
Built scalable producer-consumer architecture with Kafka, ensuring message ordering using partition keys (room-based routing)
Developed a responsive React.js frontend supporting multi-room chat and real-time message streaming
Implemented low-latency, decoupled communication between services, improving system scalability and fault tolerance

# 🚀 Kafka + Spring Boot + WebSocket Chat Project

## Complete Interview Q&A Cheat Sheet

---

## 🧠 1. Project Overview

**Q: Explain your project**
A: I built a real-time chat application using Spring Boot, Kafka, and WebSocket. The frontend sends messages via REST API, which are produced to a Kafka topic. A Kafka consumer processes them and pushes to clients using WebSocket (`SimpMessagingTemplate`). This enables scalable, decoupled, real-time communication.

---

## ⚙️ 2. Architecture & Flow

**Q: Explain end-to-end flow**
A: React → REST API (`/chat/send`) → Controller → Kafka Producer → Kafka Topic → Kafka Consumer → WebSocket (`/topic/{roomId}`) → React UI

**Q: Why REST for send and WebSocket for receive?**
A: REST is simple for sending, WebSocket enables real-time push to clients.

**Q: Can we remove Kafka?**
A: Yes, but it reduces scalability and decoupling. Kafka provides buffering, fault tolerance, and async processing.

---

## 🧠 3. Kafka Concepts

**Q: What is Kafka?**
A: Distributed event streaming platform used for high-throughput messaging.

**Q: What is a topic?**
A: Logical channel where messages are stored.

**Q: What is partitioning?**
A: Splitting topic into partitions for parallel processing.

**Q: How did you ensure ordering?**
A: Used `roomId` as key → same partition → ordered messages.

**Q: What is consumer group?**
A: Group of consumers sharing load; each partition is consumed by one consumer.

**Q: What is offset?**
A: Position of message in partition.

**Q: What happens if consumer crashes?**
A: Kafka reassigns partitions to another consumer.

**Q: Why Kafka over traditional queue?**
A: Kafka is distributed, persistent, scalable, and supports replay.

---

## 🔌 4. WebSocket & STOMP

**Q: What is WebSocket?**
A: Full-duplex communication protocol for real-time data exchange.

**Q: What is STOMP?**
A: Messaging protocol over WebSocket for pub-sub communication.

**Q: What is `/topic` vs `/app`?**
A: `/topic` → server to client, `/app` → client to server.

**Q: What is `SimpMessagingTemplate`?**
A: Sends messages from backend to subscribed WebSocket clients.

---

## 🌐 5. Frontend (React)

**Q: How does frontend receive messages?**
A: Uses STOMP client to connect to `/chat` and subscribe to `/topic/{roomId}`.

**Q: Why WebSocket over polling?**
A: WebSocket provides real-time push, lower latency, less overhead.

**Q: How did you handle multiple rooms?**
A: Each room has its own topic (`/topic/{roomId}`).

---

## ⚠️ 6. Real-World Problems

**Q: What if Kafka goes down?**
A: Messages won’t be processed; use retries, fallback, or buffering.

**Q: How do you handle duplicates?**
A: Use idempotency (unique message IDs).

**Q: How do you scale system?**
A: Increase partitions, add consumers, scale WebSocket servers.

**Q: How to persist messages?**
A: Store in DB from consumer.

---

## 🔐 7. Security

**Q: How will you secure it?**
A: Use JWT auth, secure WebSocket handshake, restrict topics.

---

## 🧱 8. System Design

**Q: How to handle millions of users?**
A: Kafka cluster, multiple consumers, load-balanced WebSocket servers.

**Q: How to support offline users?**
A: Store messages and fetch via REST on reconnect.

---

## 🧠 9. Trick Questions

**Q: Why not call consumer directly?**
A: Consumer is event-driven; Kafka decouples system.

**Q: Role of Kafka key?**
A: Determines partition and ordering.

## Kafka Retry & DLQ – Interview Q&A

**Q: What is Kafka retry?**
A: Kafka retry is a mechanism to reprocess failed messages multiple times before marking them as failed, helping handle transient errors.

---

**Q: How did you implement retry in your project?**
A: I used Spring Kafka’s `DefaultErrorHandler` with `FixedBackOff` to retry failed messages 3 times with a delay before sending them to a DLQ.

---

**Q: What is a Dead Letter Queue (DLQ)?**
A: DLQ is a separate Kafka topic where messages that fail after all retries are stored for debugging and reprocessing.

---

**Q: Why do we need DLQ?**
A: It prevents blocking the system due to bad messages and allows us to analyze and fix failures later.

---

**Q: What happens when a message fails in your system?**
A: The consumer retries processing the message 3 times. If it still fails, it is sent to a DLQ topic.

---

**Q: What is backoff in retry?**
A: Backoff is the delay between retries. In my project, I used a fixed backoff of 2 seconds.

---

**Q: What is idempotency and why is it important?**
A: Idempotency ensures that processing the same message multiple times produces the same result, preventing duplicates.

---

**Q: Can Kafka guarantee no duplicate messages?**
A: No, Kafka provides at-least-once delivery by default, so duplicates can occur. Idempotent handling is required.

---

**Q: What is the difference between retry and DLQ?**
A: Retry attempts to process the message again, while DLQ stores messages that failed after all retries.

---

**Q: How does retry improve system reliability?**
A: It handles temporary failures like network issues or service downtime without losing messages.

---

**Q: What is at-least-once delivery?**
A: It means a message is guaranteed to be delivered but may be delivered more than once.

---

**Q: What is at-most-once delivery?**
A: Message is delivered once or not at all (no retries).

---

**Q: What is exactly-once delivery?**
A: Ensures no duplicates, achieved using idempotent producers and transactional processing.

---

**Q: What challenges did you face while implementing retry?**
A: Handling duplicate messages and ensuring idempotency while maintaining message ordering.

---

**Q: How would you improve this further?**
A: I would add exponential backoff, monitoring for DLQ, and automated reprocessing of failed messages.


important - We implement retry + DLQ in the consumer because failures happen during processing, not while producing.
